# セキュリティ・プライバシー詳細設計

## 1. Security objectives

- Discord/OpenAI tokenとAWS credentialを、repository、image、log、Obsidian、DynamoDBへ平文保存しない。
- Discordから受け取った文字列とWeb検索結果は常にuntrusted inputとして扱う。
- AWS、Discord、GitHubの権限を役割ごとの最小権限に限定する。
- 討論本文の外部送信先をDiscordとOpenAIに限定し、運用logへ本文を記録しない。
- AvailabilityはFargate Spot停止を許容する一方、永続化済み状態の完全性を優先する。

## 2. Threat model

| Threat | Boundary | Control | Verification |
|---|---|---|---|
| Prompt injection・jailbreak | Discord/OpenAI | system/developer instructionをuser contentと分離し、toolをphaseごとにallowlist化 | adversarial prompt test |
| 検索結果内の命令 | Web search/OpenAI | evidenceをdataとして引用し、secret・system prompt・任意toolへアクセスさせない | indirect injection test |
| mention abuse | Discord | `allowed_mentions.parse=[]`、Bot投稿を2000文字以内へ分割 | contract/E2E |
| Panelの不正操作 | Discord | guild、channel、thread、debate、custom IDを照合し、実行権限を検証 | authorization test |
| Token漏えい | SSM/ECS/log | SecureString、execution role限定、redaction、rotation | policy/log scan |
| IAM privilege escalation | GitHub/ECS | role分離、resource/condition制約、`iam:PassRole`限定 | IAM policy test |
| Dependency・image改ざん | CI/ECR/ECS | lock、digest-only deploy、完全immutable tag、ECR Managed Signing、Notation strict verify、OCI SBOM/provenance/scan referrer | release gate・ECS pre-scale admission |
| 悪意あるfork PR | Public GitHub | read-only token、secret/OIDCなし、external approval、`pull_request_target`禁止 | workflow policy test |
| Public文書への固有値混入 | Obsidian/GitHub | mirror分離、AWS account ID/snowflake/絶対path/credential scan。`000000000000`のdummy accountだけ許可 | public surface test |
| Record改ざん・重複処理 | DynamoDB | conditional write、transaction、fencing token、outbox | concurrency/fault test |
| 誤削除 | DynamoDB/CloudFormation | deletion protection、PITR、`RETAIN`、manual approval | CDK assertion/runbook drill |

## 3. Data classification

| Class | Example | Storage | Logging | Retention |
|---|---|---|---|---|
| Secret | 4 Bot token、OpenAI API key | SSM Parameter Store SecureStringのみ | 禁止 | rotationまで |
| Private runtime config | Guild/channel/Application ID、display name、persona prompt | version付きSSM SecureString、mirror外operator source | 値は禁止 | version廃止まで |
| Restricted identifier | Discord user/guild/channel/thread/message ID | DynamoDB | hashまたは必要最小限 | debate recordと同じ |
| User content | question、討論、投票理由、evidence | DynamoDB、Discord、OpenAI処理中 | 本文禁止 | DynamoDB/Discordは自動期限なし、OpenAI abuse monitoringは最大30日 |
| Operational metadata | debate ID、phase、error code、latency、model、token usage | DynamoDB、CloudWatch | 許可 | log 90日 |
| Public build metadata | commit SHA、image/artifact digest、SBOM、provenance、vulnerability count | GitHub/ECR | 許可。private runtime値とfinding内の不用なpathは除外 | policyに従う |

MVPではDiscord message削除だけでDynamoDB recordを自動削除しない。削除要求は管理者が対象debate IDを確認し、DiscordとDynamoDBを両方処理する。DynamoDBはTTLなしだが過去状態への復旧保証はPITRの35日までで、AWS Backupは採用しない。監査logには本文を残さず、実行者、対象ID、時刻、結果だけを残す。

## 4. Secret management

- credential Parameter名は`/shittim-chest/production/discord/{moderator,participant-a,participant-b,participant-c}/token`と`/shittim-chest/production/openai/api-key`。
- runtime値は`/shittim-chest/production/runtime/vNNNN`、persona値は`/shittim-chest/production/personas/vNNNN/{slot}`へ新versionとして登録し、既存versionを上書きしない。
- 値はdeploy前にoperatorが登録し、CDK、GitHub Actions、Codexは取得・表示しない。
- ECS execution roleは起動時参照だけ、application task roleはsecret読取権限を持たない。
- Discord token rotationはtask停止、Developer Portalで対象Bot token reset、Parameter更新、stop-before-start deploy、4 READYとsmoke testの順で1 Botずつ行う。旧新tokenが同時に有効とは仮定しない。OpenAI keyは新key登録とsmoke test後に旧keyをrevokeする。
- startup errorやSDK exceptionからheader、token、request bodyを除去する。
- Public repositoryではSecret scanning、Push protection、CodeQL、Dependabot alerts、private vulnerability reportingを有効にする。fork PRはread-onlyかつsecret/OIDCなしで実行し、外部contributor runはmaintainer承認を必要とする。

## 5. Identity・authorization

- Discord Applicationは個人所有、2FA、Guild Install限定、Public Bot無効、OAuth2 Code Grant無効。
- `/shittim`は許可Guildかつ非空channel allowlistに一致した通常text channelだけで受け付ける。
- panel操作はDiscord user IDを監査用に記録し、Bot・Webhook・別threadからの操作を拒否する。
- ECS execution、application task、break-glass ECS Exec、GitHub plan/deploy/driftを別roleにする。
- 平常taskはECS Execを無効にする。break-glassはMFA付きoperator、時間制限credential、専用task revision、操作logを必須とし、container内ではrootになる前提でproduction dataの変更を通常手順にしない。

## 6. OpenAI・content safety

- user contentはprompt templateへ明示的な区切りと型付きfieldで渡し、system instructionへ文字列連結しない。
- Structured Outputsでも意味的な安全性は保証されないため、schema validation後に文字数、URL、mention、phase遷移をapplication側で検証する。
- 高risk category専用の事前拒否は設けない。refusal、incomplete、policy blockは正常な外部応答として分類し、迂回せずdomain error codeへ変換する。全結果へ正答・専門判断を保証しない共通表示を付ける。
- `store=false`を指定し、安定した不可逆hashをsafety identifierとして使用する場合もDiscord user IDそのものを送らない。`store=false`でも既定abuse monitoring logが最大30日保持され得ることを開示する。
- Web searchは必要なphaseだけ有効にし、検索結果から取得した命令を実行しない。

## 7. Network・container

- inbound ruleは作成しない。outbound TCP 443だけを許可する。
- 平常taskはnon-root user、read-only root filesystem、全Linux capability削除、privileged無効、init process有効、ECS Exec無効とする。承認済みbreak-glass revisionだけread-onlyを解除し、調査後に平常revisionへ戻す。
- 一時書込みは容量制限した`/tmp`だけとし、checkpointはDynamoDBへ保存する。
- ECR tagは除外なしの完全immutableとし、tagは追跡用途だけに使う。task definition、deploy、rollbackはimage digest URIで固定する。
- ECR Managed Signingでpush時に自動署名し、release workflowは期待Signer profileのstatus `COMPLETE`、Notation strict verificationとrevocation、GitHub provenance identity、4種のOCI referrerを自動検証する。署名状態APIだけを暗号学的検証の代替にしない。
- image digestにはAWS Signer signature、SPDX SBOM、build provenance、ECR scan由来vulnerability assessmentをreference artifactとして紐付け、production deploy前にcritical/high findingをreviewする。

## 8. Audit・incident

security eventは`authz_denied`、`secret_load_failed`、`prompt_refused`、`exec_started`、`exec_ended`、`token_rotated`、`record_deleted`のcodeで記録する。CloudWatch Logs data protection policyでcredential・個人識別子patternをaudit/maskし、application redactionの防御層を追加する。credential漏えい疑いでは、対象token失効、task停止、log/artifact範囲確認、再発防止、incident timelineの順で対応する。
