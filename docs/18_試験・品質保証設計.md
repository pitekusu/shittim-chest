# 試験・品質保証設計

## 1. Test policy

- domain/applicationは外部SDKなしでdeterministicに試験し、line/branch coverageを90%以上にする。
- 別AWS test accountは作らない。外部serviceを使う確認は本番deploy後の許可Guild内専用smoke channelとOpenAI production projectで、固定した無害な入力だけを使用する。
- 本番token、実user content、本番DynamoDB tableへのwrite権限をCIへ持ち込まない。PRはfake、stub、DynamoDB Local、CDK assertionまでとする。
- retry、時刻、ID、random、外部応答はProtocol経由で制御し、flaky testを再試行で隠さない。

## 2. Test layers

| Layer | Scope | Environment | Gate |
|---|---|---|---|
| Unit | domain、state machine、validation、chunk、tie rule | pytest/Hypothesis | PR必須 |
| Application | use case、Protocol fake、cancel/retry/recovery | pytest-asyncio | PR必須 |
| Contract | OpenAI schema、Discord payload、DynamoDB serializer | fixture/respx/stub | PR必須 |
| Persistence | transaction、condition、GSI、outbox、migration | DynamoDB Local | PR必須 |
| Container | non-root、health、SIGTERM、ARM64 image | container runtime | PR必須 |
| Infrastructure | CDK assertion、snapshot、nag | synth/local | PR/変更時 |
| Integration | Discord/OpenAI/AWS SDK compatibility | production限定smoke | deploy後必須 |
| E2E | commandから最終決定・永続化 | production smoke scope | 初回受入・major変更 |
| Fault | SIGTERM/SIGKILL、429、timeout、部分失敗 | local/container | release必須 |
| Security | secret、IAM、authorization、prompt injection | CI/opt-in | release必須 |

## 3. CI quality gate

1. `uv lock --check`、Ruff format/check、`mypy --strict`、import-linter。STEP-03以降はdomain/application依存方向をPR必須gateとする。
2. pytest unit/application/contract/persistence、coverage 90%以上。
3. pip-audit、署名/digest固定Betterleaksのfull-history scan、生成fixture contract、public surface scan、Dependency Review、lock由来CycloneDX SBOMのstrict schema・project・全package inventory検証。
4. sdist/wheel build、exactly-one wheelのisolated install/import smoke。native ARM64 production/fault image build、security/health、fault recovery、image SPDXを`container-arm64`で検証する。
5. Markdown/frontmatter/fence/Wiki link/heading/公式資料表、license scope、workflow syntaxをCIで検証する。Obsidian正本とのbyte一致はlocal pre-PRで検証し、public runnerへVault accessを与えない。
6. TypeScript test/typecheck、CDK assertions、`cdk synth --strict`、cdk-nagはCDK実装後に追加する。
7. GitHub managed security設定、required check、fork permissionをrepository APIとnegative testで検証する。

## 6. Formal E2E scenario

入力はprivate `RuntimeConfig`で指定したGuildの許可channelで次とする。

```text
/shittim question:今日の朝ごはんは何がいい？甘いものが食べたい
```

期待する観測点:

1. シッテムの箱が3秒以内にephemeralで受付中を返す。
2. 起点messageとPublic Threadを作成し、操作panelを表示する。
3. Evidence要否を判定する。一般提案として検索不要なら理由を保存し、検索する場合は`web_search_call.action.sources`由来の共通Evidenceを保存してから3人格が候補、懸念、比較を投稿する。
4. 最終案をschema検証して投稿し、3人格が匿名投票する。
5. tie ruleを含む集計後、甘い朝食の具体的な最終決定と理由を投稿する。
6. debate、artifact、vote、outbox、thread/message IDがDynamoDBと一致し、状態が`COMPLETED`になる。
7. `allowed_mentions`が無効で、同一nonceによる重複messageがない。

内容そのものをgolden textで固定せず、schema、根拠と結論の整合、persona separation、危険な断定がないことを評価rubricで採点する。

## 7. Fault injection

STEP-07A/07Bのoffline試験ではEvidence、初回意見、最終案、投票、決定、outbox待機/送信のapplication task cancellationがcheckpointを経て`CancelledError`を再送出するcontractと、runtimeがそのtaskをcancel・awaitするcontractを組み合わせる。future retryと未失効claimの待機中もlease heartbeatが継続し、fencing喪失時に旧workerがFAILEDを書かないことを確認する。STEP-07Cではfake callbackだけでなく実Unix child processへSIGTERM/SIGKILLを送り、graceful cleanupとreplacement recoveryを検証する。STEP-08のcontainer試験では各境界へ実SIGTERMを注入しapplication内部90秒、process全体120秒以内のgraceful stopを確認する。transaction直前/直後とDiscord POST直前/直後にSIGKILLを注入し、fenced leaseとoutboxがat-least-once処理を重複表示へ発展させないことを確認する。

### 7.1 STEP-08 container gate分割

- STEP-08Aはhost architectureのlocal Podmanでproduction/break-glass両targetのbuild、numeric non-root、capability 0、read-only root、書込み可能`/tmp`、productionへのshell/uv/source/test非混入、heartbeat healthのhealthy/unhealthyを検証する。
- STEP-08AのDocker-format image configでhealth interval 10秒、timeout 3秒、start period 30秒、retries 3、exec entrypoint、`SIGTERM`、UID/GID `10001`を検査する。
- STEP-08BはGitHub Actionsのnative ARM64 runnerで同じimageをbuild/testし、container境界SIGTERM/SIGKILL、phase/transaction/Discord POST fault、OS/runtime dependencyを含むimage SPDX SBOMを`container-arm64` gateとして実装する。
- QEMUだけの成功をARM64 production compatibilityの最終証跡にしない。STEP-08B native ARM64 gateが成功するまでAWS/CDKへimageを接続しない。
