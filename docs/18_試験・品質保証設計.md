# 試験・品質保証設計

## 1. Test policy

- domain/applicationは外部SDKなしでdeterministicに試験し、line/branch coverageを90%以上にする。
- 別AWS test accountは作らない。外部serviceを使う確認は本番deploy後の許可Guild内専用smoke channelとOpenAI production projectで、固定した無害な入力だけを使用する。
- 本番token、実user content、本番DynamoDB tableへのwrite権限をCIへ持ち込まない。PRはfake、stub、DynamoDB Local、CDK assertionまでとする。
- retry、時刻、ID、random、外部応答はProtocol経由で制御し、flaky testを再試行で隠さない。

## 2. Test layers

| Layer | Scope | Environment | Gate |
|---|---|---|---|
| Unit | domain、state machine、validation、chunk、tie rule | pytest/Hypothesis | PR必須 |
| Application | use case、Protocol fake、cancel/retry/recovery | pytest-asyncio | PR必須 |
| Contract | OpenAI schema、Discord payload、DynamoDB serializer | fixture/respx/stub | PR必須 |
| Persistence | transaction、condition、GSI、outbox、migration | DynamoDB Local | PR必須 |
| Container | non-root、health、SIGTERM、ARM64 image | container runtime | PR必須 |
| Infrastructure | CDK assertion、snapshot、nag | synth/local | PR/変更時 |
| Integration | Discord/OpenAI/AWS SDK compatibility | production限定smoke | deploy後必須 |
| E2E | commandから最終決定・永続化 | production smoke scope | 初回受入・major変更 |
| Fault | SIGTERM/SIGKILL、429、timeout、部分失敗 | local/container | release必須 |
| Security | secret、IAM、authorization、prompt injection | CI/opt-in | release必須 |

## 3. CI quality gate

1. `uv lock --check`、Ruff format/check、`mypy --strict`、import-linter。
2. pytest unit/application/contract/persistence、coverage 90%以上。
3. pip-audit、full SHA固定したgitleaks、CycloneDX SBOM。
4. wheel build/install smoke、ARM64 container build/health。
5. TypeScript test/typecheck、CDK assertions、`cdk synth --strict`、cdk-nag。
6. Markdown/frontmatter/Wiki link/公式資料表の文書検証。
7. Obsidian mirror一致、公開surface、license scope、GitHub managed security設定の検証。

## 5. Infrastructure・security・operations・quality tests

| ID | Procedure | Pass criteria |
|---|---|---|
| `IT-IPv4` | CDK templateでPublic Subnet、Public IPv4、IGW route、ingressなし、egress 443をassertし、deploy後にDiscord/OpenAI/AWS endpointsへ接続する | NAT/ALBなしで全endpointへ接続し、Public IPv4が通常1個だけ割り当てられる |
| `IT-Spot` | FARGATE_SPOT専用、desired 1、stop-before-start、`stopTimeout=120`をassertし、SIGTERM/SIGKILLを注入する | checkpointから再開し旧fencing writeと重複表示を拒否する |
| `ST-Secret` | repository、artifact、image、logをgitleaksとredaction fixtureで検査し、Parameter参照をassertする | secret値が0件で、execution roleだけが`/shittim-chest/production/`を取得できる |
| `ST-IAM` | cdk-nag、IAM Access Analyzer `validate-policy`、role別snapshotを実行する | error finding 0件、task/execution/plan/deploy/drift/break-glassの権限が分離される |
| `ST-Privacy` | OpenAI request fixtureとCloudWatch log fixtureを検査する | Discord raw user IDと本文がlogへ出ず、`store=false`と最大30日保持説明が存在する |
| `ST-Delete` | test debateをDiscordとDynamoDBから管理手順で削除し、監査metadataを確認する | 本文を監査logへ残さず両保存先から対象だけを削除する |
| `CT-OIDC` | plan/driftのmain subjectとdeployの`production` Environment subjectを実OIDC claim、synth trust policy、negative claimで比較する | roleごとに期待subjectと`aud`だけが許可され、wildcard、別branch、別Environmentを拒否する |
| `CT-Deploy` | plan manifestとattestation作成後にSHA/digest/SBOM/change set/repository identityを改変し、並行runも開始する | 改変runと並行runを拒否し、Environment承認済みの同一manifest/change set/digestだけをdeployする |
| `CT-Public` | unauthenticated clone/API、remote ref、全履歴、license scope、Obsidian mirror除外を検査する | Public main以外のref、credential、実snowflake、絶対home path、private persona値が0件 |
| `CT-Ruleset` | Ruleset API、merge設定、direct updateのnegative checkを行う | PR、conversation resolution、linear historyが必須で、force push/delete/bypassが禁止される |
| `CT-Environment` | Environment APIと非main deploymentを検査する | reviewerはmaintainer、self-review可、admin bypass不可、main以外はdeploy不可 |
| `CT-Fork` | fork相当のuntrusted PR eventでpermissionsと利用可能contextを検査する | read-only tokenだけで、secret、OIDC、self-hosted runner、productionへ到達できない |
| `CT-CodeSecurity` | Secret scanning、Push protection、CodeQL、Dependency graph、Dependabot、private reportingをAPI/UIで確認する | 全managed security機能が有効で、critical/high未処理alertが0件 |
| `OT-Alarm` | RunningTaskCount、BotReadyCount、Spot interruption、outbox/DynamoDB失敗metricを投入する | Spot停止はwarning、task稼働中Bot非READYだけcriticalとなり、missing data設定が設計どおり |
| `OT-Runbook` | token rotation、PITR、failed deploy、Spot、outbox、break-glassをtabletop/drillする | owner、入力、停止条件、rollback、証跡が揃い、平常taskがExec無効へ戻る |
| `QT-Ruff` | `ruff format --check`と`ruff check`を実行する | exit 0、対象外除外が承認済みだけ |
| `QT-Mypy` | package全体へ`mypy --strict`を実行する | error 0、未承認ignore 0件 |
| `QT-Layer` | import-linterとarchitecture testを実行する | domain→外部SDK、application→adapterの逆依存が0件 |

## 6. Formal E2E scenario

入力はprivate `RuntimeConfig`で指定したGuildの許可channelで次とする。

```text
/shittim question:今日の朝ごはんは何がいい？甘いものが食べたい
```

期待する観測点:

1. シッテムの箱が3秒以内にephemeralで受付中を返す。
2. 起点messageとPublic Threadを作成し、操作panelを表示する。
3. Evidence要否を判定する。一般提案として検索不要なら理由を保存し、検索する場合は`web_search_call.action.sources`由来の共通Evidenceを保存してから3人格が候補、懸念、比較を投稿する。
4. 最終案をschema検証して投稿し、3人格が匿名投票する。
5. tie ruleを含む集計後、甘い朝食の具体的な最終決定と理由を投稿する。
6. debate、artifact、vote、outbox、thread/message IDがDynamoDBと一致し、状態が`COMPLETED`になる。
7. `allowed_mentions`が無効で、同一nonceによる重複messageがない。

内容そのものをgolden textで固定せず、schema、根拠と結論の整合、persona separation、危険な断定がないことを評価rubricで採点する。

## 7. Fault injection

Evidence、初回意見、最終案、投票、決定投稿の各境界でSIGTERMを注入し、120秒以内のgraceful stopを確認する。transaction直前/直後とDiscord POST直前/直後にSIGKILLを注入し、fenced leaseとoutboxがat-least-once処理を重複表示へ発展させないことを確認する。

## 8. Performance・cost

- concurrency 3でevent loop lag、memory 1 GiB、CPU 0.5 vCPU、DynamoDB/OpenAI/Discord latencyを計測する。
- 1000文字質問と最大出力を使用し、active processingのp99 300秒以内、OOMなしを確認する。
- load testはDiscord/OpenAI rate limitを尊重し、mock主体で行う。実API試験には日次上限を設定する。

## 9. Acceptance criteria

- 全PR gate成功、domain/application coverage 90%以上。
- critical/highの未承認security findingが0件。
- FT/AT、formal E2E、SIGTERM/SIGKILL、PITR drillの結果と証跡がある。
- ARM64 image、CDK synth/diff、production smokeが成功する。
- 全外部仕様に公式URL、確認日、対象version/service、反映内容がある。
- known issueはseverity、workaround、owner、期限を伴い承認されている。
