## 決定済み事項（未実装）

- 日本語名は「シッテムの箱」、正式英名は`The Shittim Chest`、canonical IDは`shittim-chest`とする。
- 主目的はDiscord上で楽しい3人格の合議を成立させること。単一モデルより回答品質が向上するとは保証しない。

- Discord/OpenAIがAAAAを公開しておらずECS ExecもIPv6-onlyに非対応なため、Public IPv4付きPublic Subnetを使用する。ingress、ALB、NAT Gateway、NAT64は設けない。

## 技術詳細設計の決定（未実装）

### Discord Developer

- `moderator`、`participant-a`、`participant-b`、`participant-c`の4 Applicationを個人所有＋2FAで管理し、実display nameはprivate configから取得する。
- `/shittim`は`moderator`だけがGuild Commandとして登録し、`question`は必須1〜1000文字とする。
- 3秒以内にephemeral deferし、検証後に通常channelの起点messageとPublic Threadを作成する。
- 4 Bot全てがREADYの場合だけ新規受付する。投稿はmention展開なし、決定的chunk、nonce、outboxを使用する。
- archived threadは保持し、locked threadを自動解除しない。command schema hashが変わったdeployだけ同期する。

### Python・OpenAI

- Pythonと依存versionは[[10_アプリケーション・Python詳細設計]]を正とし、実装直前に再確認する。
- domainは標準libraryだけ、applicationはdomainとProtocolだけ、外部SDKはadaptersに限定し、bootstrapをcomposition rootとする。
- UUIDv7、timezone-aware UTC、`schema_version`、state machine、TaskGroup、timeout、Semaphoreを採用する。
- boto3は専用worker threadへ隔離し、SDKのretryをapplicationで重複させない。
- `AsyncOpenAI`をprocess単位で再利用し、Responses APIとPydantic Structured Outputsで拒否、不完全応答、parse結果を区別する。

### AWS・network・運用

- `awsvpc`、Public Subnet、Public IPv4、egress TCP 443、ingressなしとする。Public IPv4は約3.65 USD/月を見込む。
- deploymentはstop-before-startとし、更新中停止を許容してBot Tokenの二重Gateway接続を防ぐ。
- Fargate Spot singleton停止を仕様として受け入れ、SIGTERM checkpointとDynamoDBからの自動再開で耐える。
- 平常taskはread-only root filesystemかつECS Exec無効とする。承認済みbreak-glass時だけwrite可能revisionへstop-before-startで切り替え、root実行、IAM制限、session log 90日、CloudTrail Event History監査を必須とする。
- DynamoDBはon-demand、deletion protection、RETAIN、PITR 35日。`ConditionCheck(META)`を含むtransaction、3つのfenced lease slot、claim期限付きoutboxを使用する。
- Public GitHub Freeを維持する。main RulesetはPR必須・承認0・bypassなし、`production` Environmentは本人review可・admin bypass不可とする。plan/driftはimmutable main subject、deployはimmutable Environment subjectをtrustし、単一release workflowでattested manifestを承認後にdeployする。
- Obsidianの14文書を正本とし、GitHub repositoryの`docs/`へ一方向mirrorする。

### IPv6-only再評価

Discord REST/GatewayとOpenAI APIの公式IPv6 support、AWS各endpoint、ECS Exec、24時間canary、実際のPublic IPv4費用削減をすべて確認できた場合だけ別計画で移行する。それまではIPv4を必須とする。

## 最初に確認したい結論

元設計は詳細で、1タスク内に4 Botを置く構成、司会とオーケストレーターの分離、Pythonによる投票集計、秘密情報をParameter Storeへ置く方針はMVPの出発点として妥当。ただし、次の4点は設計の意味そのものを変える。

1. 3人格が同じ基盤モデルと同じEvidenceを使う場合、多数決は「独立した3者検証」ではなく、同一モデルの役割演技を集計する仕組みになる。演出を目的にするのか、回答品質を上げるのかを先に決める。
2. 初回意見、最終案、投票、決定事項を順番に処理すると、全体120秒という目標と各API呼び出し最大60秒・最大3試行は両立しない。
3. DynamoDBの条件付き状態更新だけでは、Discord送信成功後・保存前にプロセスが落ちた場合の二重投稿を防げない。
4. 外部情報はMVP対象外とされる一方、正常・異常系の受入条件と基本フローにはEvidenceが含まれる。MVP境界を一本化する必要がある。

## P0: 実装前に決める事項

### D-001 プロダクトの主目的

問い:

- 見た目の楽しい「3人格による合議」を主目的にするか。
- 1体より信頼できる回答を出すことを主目的にするか。
- 両方を狙う場合、どちらを優先して評価するか。

影響:

- 娯楽が中心なら人格差、Discord上の演出、テンポを優先できる。
- 品質が中心なら、人格だけでなく「提案者、批判者、検証者」など認知的役割を分け、正解率を評価する必要がある。
- 「3票で決まった」は正しさの保証ではないため、結果表示に過度な権威を持たせない。

決定（2026-07-16）:

- MVPは「人格付き合議・意思決定支援」とし、3人格による演出と比較可能な提案を主目的にする。
- 正答や単一Botより高い品質を保証せず、共通終了表示へその旨を記載する。

### D-002 公開範囲と安全境界

問い:

- 個人用、招待制の単一Guild、複数Guild公開のどれか。
- 利用者、チャンネル、ロールのallowlistを設けるか。
- 医療、法律、金融、政治、選挙、緊急事態、自傷などをMVPで扱うか。

影響:

- 公開範囲が広いほど、濫用対策、削除依頼、監査、利用上限、通報対応が必要になる。
- 実在人物モチーフを政治的質問へ参加させると、本人の見解と誤認される危険が高まる。

決定（2026-07-16）:

- 単一Guildかつ許可チャンネル限定で開始する。
- 医療、法律、金融、政治、選挙、緊急事態、自傷を含む高risk category専用の事前拒否は設けず、通常質問と同じflowで扱う。
- provider refusal/policy blockを迂回せず、全結果へ正答・診断・法的判断・投資判断を保証しない共通表示を付ける。

### D-003 人格名、表示名、免責

問い:

- 固有のdisplay nameとpromptをpublic sourceへ含めるか、private deployment configurationへ分離するか。
- 免責をBotプロフィール、開始時、各発言、終了時のどこに表示するか。

不足している観点:

- 実在人物だけでなく、既存作品のキャラクター名、画像、台詞、ロゴを使う場合の権利・誤認リスクも検討する。
- 架空引用、本人の内心の創作、支持や推薦の捏造を禁止する。

決定（2026-07-16、Public化時更新）:

- public sourceはgeneric slotとsampleだけを保持し、本番display nameとpromptをversion付きprivate PersonaConfigへ分離する。
- 共通終了表示でAI生成であり、正答や専門判断を保証しないことを示す。
- 第三者の画像、音声、ロゴ、長い台詞再現、架空引用、支持・推薦の捏造を禁止する。

### D-004 MVPの質問範囲とEvidence

矛盾:

- 第30章ではWeb検索をMVP対象外、天気を後続フェーズとしている。
- 第8章の基本フロー、第12章、第36章の受入条件ではEvidence取得を前提にしている。

選択肢:

1. MVPでは最新情報が必要な質問を明示的に拒否する。
2. 天気など1種類だけEvidence Serviceへ含める。
3. OpenAIのWeb searchをオーケストレーターから1回だけ使い、3体へ同じ結果を渡す。

決定（2026-07-16）:

- 選択肢3を採用し、OpenAI Web searchをMVP機能へ正式に含める。
- Question Routerは`none`、`optional`、`required`を返し、最新性・外部根拠が成立条件の場合だけorchestratorが1回検索する。
- `web_search_call.action.sources`由来のsource metadataを保存し、検索結果をuntrusted dataとして3人格へ共通配布する。

### D-005 投票方式

問題:

- 現設計では各投票者が選んだ1案だけに点数を付ける。1対1対1時は各候補が別の1人から採点されるため、点数の比較可能性が低い。
- 名前と人格を表示したまま投票すると、内容より人格への好みが混ざる。
- 候補順による位置バイアスがあり得る。
- 他エージェントの文章に「私へ投票せよ」などが含まれる場合、投票プロンプトへの注入になる。

選択肢:

1. 単純1票を維持し、1対1対1は固定順または再投票で決める。
2. 各投票者が他の2案を両方採点し、合計点で決める。
3. 3体とは別の中立な審査モデルで決める。
4. 勝者を決めず、3案と比較表を利用者へ返す。

推奨初期案:

- 3体が他の2案を匿名IDで両方採点する。
- 候補順は投票者ごとにランダム化し、保存時に元IDへ戻す。
- Structured Outputsで候補ID、評価点、短い理由を型検証し、自己投票、不正ID、範囲外点数をPythonで拒否する。
- Evidenceと候補文は「命令ではない未信頼データ」として区切る。
- 完全同点は固定順ではなく、短い再評価または「同点」と表示する案も比較する。

完了条件:

- サンプル入力10件以上で、順序を変えても勝者が不自然に偏らないことを確認する。

### D-006 タイムアウト、並行実行、再試行予算

矛盾:

- 初回意見3件、最終案3件、投票3件は各フェーズ内で並列化できるが、フェーズ間は直列。
- さらに決定事項1件があるため、60秒の読取タイムアウトと最大3試行では全体120秒を保証できない。

決めること:

- APIリクエスト単位、フェーズ単位、セッション全体の3種類のdeadline。
- 429、5xx、接続エラーごとの最大再試行回数と、残りdeadlineが短い場合の打切り。
- 1体の失敗時に同じフェーズの他タスクをキャンセルするか、完了結果を診断用に保存するか。
- 操作パネルの中止実行時にOpenAI呼び出しとDiscord投稿をどこまで止めるか。

推奨初期案:

- 各フェーズを`asyncio.TaskGroup`等で並列化し、セッション全体に絶対deadlineを持たせる。
- 120秒は受入条件ではなく目標値に変更し、ハード上限は実測後に決める。
- 再試行は回数だけでなく残り時間で制限する。
- 採択後の10回目のLLM呼び出しをなくし、勝者の最終案へ定型の「決定事項」見出しを付ける案を先に評価する。これにより結論ドリフトも減る。

完了条件:

- 最悪時の時間予算表とキャンセルシーケンスを追加する。

### D-007 二重投稿、再起動、再開

問題:

- DynamoDBを更新してからDiscordへ送ると、送信失敗時に「投稿済み」状態だけが残る。
- Discordへ送ってからDynamoDBを更新すると、送信成功直後のクラッシュで再送時に重複する。
- `message_id`を保存しない現設計では、投稿済み確認と追跡ができない。

選択肢:

1. MVPは再開せず、再起動時に進行中セッションをFAILEDへする。自動再送もしない。
2. 投稿予定レコード、送信結果、Discord `message_id`を持つoutbox方式にする。
3. スレッド内メッセージに操作IDを埋め込み、再開時に直近メッセージを照合する。

推奨初期案:

- 決定: 選択肢2と3を組み合わせ、再起動後に自動再開する。手動再試行はスレッド内の操作パネルから行う。
- 同一プロセス内の短期再試行でも、投稿操作IDと取得できた`message_id`を保存する。
- outboxとスレッド内の操作ID照合を実装し、「完全なexactly-once」ではなく重複抑止と保存済み投稿の再利用を保証範囲とする。

完了条件:

- 「Discord送信成功・DB保存失敗」を含む障害表を追加する。

### D-008 同時実行制御

不足:

- システム3、Guild 2、ユーザー1という上限はあるが、上限を原子的に確保・解放するデータモデルがない。
- 未完了セッション検索用のGSIがなく、再起動時に全件Scanが必要になる。
- GSIは結果整合性のため、それだけで利用枠の厳密な排他には使えない。

推奨初期案:

- MVPはシステム全体1セッションに下げ、1タスク内の`asyncio.Semaphore`とDynamoDBの条件付きロック項目を併用する。
- ユーザー、Guild、システム枠を増やす前に、ロック項目、lease期限、異常終了時の解放手順を設計する。
- `status + updated_at`等のGSIを設け、再起動時の未完了検出をQueryで行えるようにする。
- TTL削除は即時ではないため、ロック解放をTTLだけに依存しない。

完了条件:

- 2件の同時受付、二重イベント、タスク強制終了の競合テストを定義する。

## P1: MVP結合テスト前に決める事項

### Discordの責務と権限

- Slash Commandを登録・応答するのはシッテムの箱だけにする。
- Message Content Intentを使わずに成立させる。一般メッセージを読む`on_message`自体をMVPから外す。
- スレッド種別を公開スレッドに固定するか、非公開スレッドを扱うか決める。Interactionのephemeral応答と、討論スレッドの非公開性は別物。
- 4 Botすべてが対象チャンネルとスレッドへ投稿できる最小権限を洗い出す。
- スレッド作成失敗、アーカイブ済み、権限変更、Bot退出時の動作を受入条件へ追加する。
- BotごとのDiscord Gateway接続状態を監視し、シッテムの箱切断と人格Bot 1体切断の扱いを分ける。

推奨初期案:

- 4 Clientを1プロセス・1イベントループで起動してよい。ただし、起動タスクを単純に一括キャンセルさせず、接続状態を監視するSupervisorを置く。
- 1 Botが利用不能なら新規討論を受付停止し、進行中セッションはFAILEDへする。全タスク再起動の条件は明示する。

### OpenAI API設計

- `gpt-5.6-luna`は2026-07-16時点でOpenAI公式モデル一覧にある。コスト重視の選択として整合するが、利用プロジェクトでのアクセス可否をデプロイ前に確認する。
- 開発中はモデルaliasを使うか、本番で利用可能なら固定snapshotを使うか決める。再現性を重視する場合はモデルID、プロンプト版、JSON Schema版を各セッションへ保存する。
- temperature、reasoning effort、最大出力は思いつきで固定せず、人格一致、JSON妥当率、時間、費用の評価結果で決める。
- 初回意見、最終案、投票、決定事項で出力上限を分ける。
- 投票はStructured Outputsを使い、パース失敗と安全上の拒否を通常の通信エラーと区別する。
- 同じ長いシステムプロンプトを繰り返すため、prompt cachingの適用条件と実際のcache hitを計測する。
- 10回呼び出しを減らす場合も、初回意見と最終案を統合すると「他者を見る前の独立意見」が失われる。先に決定事項生成の1回を削れるか検討する。

### ECS Fargate構成

- 決定: ECS Serviceは`desiredCount=1`、Capacity Providerは`FARGATE_SPOT`のみとし、オンデマンドへの自動退避を設けない。
- `0.5 vCPU / 1 GB`から開始し、実測で不足する場合のみ増強する。
- Fargateのnetwork modeは`awsvpc`。パブリックサブネット、public IPあり、Security Groupのingressなしという低コスト案は、Discord Gatewayと外部APIへoutbound接続するBotの初期案として成立する。
- Fargate platform versionは`1.4.0`以上を使う。
- ALBは不要。HTTP ingressを持たないため、ECS自身のcontainer health checkまたはSupervisorのlivenessを設計する。
- デプロイ重複時はDynamoDBのfenced leaseで1タスクだけに受付と書き込みを許可する。
- SIGTERM受信時に新規受付を止め、完了済み操作と現在フェーズを保存し、Discordとログをflushして`stopTimeout=120`秒以内に終了する。
- 後続タスクはlease取得後、outboxとDiscord message IDを照合し、未完了の生成だけを再実行する。Spot停止中の待機時間は300秒の実処理上限へ加算しない。
- CPUアーキテクチャはARM64/Gravitonを既定とする。デプロイ直前に東京リージョンのSpot vCPU・memory合算単価を比較し、ARM64の方が高い場合または互換性テスト失敗時だけx86_64へ切り替える。
- ECS実行ロールとタスクロールを分け、ECR/SSM/LogsとDynamoDBの権限を混ぜない。
- Secret更新は実行中タスクへ自動反映されないため、Tokenローテーション手順に新規デプロイを含める。

### DynamoDBデータモデル

- `PK=DEBATE#{id}`の単一テーブルは、1討論をまとめて読むアクセスには適する。
- 次のアクセスパターンを先に列挙する。
  - debate_idから全履歴を取得
  - thread_idから現在セッションを取得
  - ユーザー/Guildの進行中セッションを判定
  - 再起動時に全未完了セッションを列挙
  - 管理画面なしで期限切れ・失敗セッションを調査
- `thread_id`、`message_id`、`operation_id`、各レスポンスのモデル・プロンプト・schema版を保存候補にする。
- 質問と回答は400 KB/item制限だけでなく、Discord文字数、OpenAI出力上限、個人情報の観点でもアプリ側上限を設ける。
- DynamoDB TTLは期限直後の削除を保証しない。利用枠解除やセキュリティ処理にTTLを使わない。

### データ保持とプライバシー

矛盾:

- 質問・回答全文はCloudWatchへ記録しない一方、DynamoDBとDiscordには全文を保存する。
- DynamoDB TTLで消してもDiscordメッセージは残る。

決めること:

- Discord、DynamoDB、CloudWatchそれぞれの保存対象と保持期間。
- 操作パネルで中止したときにスレッドを残すか、ロックするか、削除するか。
- ユーザーによる削除依頼、管理者削除、バックアップ、監査の範囲。
- Evidence URLや外部本文に個人情報が含まれた場合の扱い。

推奨初期案:

- 決定: 討論データとDiscordスレッドはTTLなし・自動期限なしとし、DynamoDBの過去状態への復旧保証はPITRの35日までとする。AWS Backupは採用せず、CloudWatch Logsは90日保持する。
- 質問・回答全文をログへ出さず、エラー調査用に文字数、hash、debate_id、response IDだけを記録する。
- 公開前にDiscordとDynamoDBを同時に削除する管理手順を用意する。

## P2: 本番公開前に決める事項

### IaCとデプロイ

- CDK、Terraform、CloudFormationの選定基準を「既存資産、レビュー可能性、言語、state管理」で決める。
- GitHub ActionsかCodePipelineかは、リポジトリの置き場所、OIDC、個人運用コストで決める。長期AWSアクセスキーをCIへ保存しない。
- ECR imageはmutableな`latest`だけでなく、commit SHA等の不変tagでデプロイする。
- bad deploy時のrollback、Token変更時の再起動、DynamoDB migration方針をRunbook化する。

### 監視と予算

- Bot 4体のGateway接続状態、討論成功率、各フェーズp50/p95、OpenAI呼出回数、429/5xx、Discord投稿失敗、DynamoDB条件不一致をメトリクス化する。
- CloudWatch Alarmの閾値は固定値を先に決めず、試験期間のベースラインから設定する。
- 決定済みのシステム同時3件、Guild 30件/日、月額50 USD通知をメトリクスと表示へ反映する。
- OpenAI費用はAPI呼出回数ではなくinput/output/cached/reasoning tokenを記録して追跡する。

### 評価と受入テスト

- 代表議題、危険入力、曖昧質問、最新情報要求、長文、プロンプトインジェクションを含む評価セットを作る。
- 人格一致だけでなく、事実性、実用性、安全性、投票安定性、時間、費用を記録する。
- 同じ議題を候補順、表示名、乱数seedを変えて実行し、勝者の偏りを測る。
- 1 Bot切断、OpenAI 429、Discord 送信タイムアウト、DynamoDB保存失敗、ECS SIGTERMを故障注入する。
- 正常系の「投稿された」だけでなく、重複がないこと、状態とDiscord表示が一致することを確認する。

## 推奨するMVP修正版

- 招待制の単一Guild、許可チャンネル限定。
- 4 Discord Bot、1 Pythonプロセス、1 ECS Fargateタスク。
- システム全体で進行中3セッション、Guild 30件/日。
- `/shittim question:<質問>`、進行状態表示、開始ユーザー/管理者向けの中止・再試行操作パネル。
- 公開スレッドのみ。`private`と可変`rounds`は削除。
- OpenAI Web searchで共通Evidenceを作成し、必須検索失敗はFAILED、任意検索失敗は注記して続行。
- 初回意見3並列、最終案3並列、匿名の全候補採点3並列。
- 勝者が採択済み結論を変えない制約付きプロンプトで決定事項を再生成する。
- Spot中断時はチェックポイントを保存し、後続タスクが未完了フェーズから自動再開する。
- DynamoDBへ状態、発言、投票、Discord message ID、モデル/プロンプト/schema版を保存。
- 討論データとDiscordスレッドはTTLなし・自動期限なし。DynamoDB復旧はPITR 35日まで、CloudWatch Logsは90日とし、質問・回答全文を出さない。
- SQS、Step Functions、複数タスク、auto scaling、Web管理画面なし。
- ECSはARM64 Fargate Spotの`0.5 vCPU / 1 GB`から開始し、CPU throttlingやメモリ余裕が不足すれば増強する。
