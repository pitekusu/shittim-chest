# 運用保守・監視・障害対応設計

## 1. Service objective

MVPに外部SLAは設定しない。Fargate Spot容量不足による受付停止は許容し、on-demandへfailoverしない。正常起動中は4 Bot READY、受付済みdebateの再開可能性、永続化と投稿の整合性を運用品質の中心とする。

目標値は月次で、受付済みdebate成功率95%以上、application起因の失敗率5%未満、active processingのp99 300秒以内とする。Spot容量不足時間は別集計し、application SLOから除外するが報告には含める。

## 2. Structured logging

JSON logの共通fieldは`timestamp`、`level`、`event`、`service`、`version`、`debate_id`、`phase`、`attempt`、`duration_ms`、`error_code`、`aws_request_id`とする。Discord/OpenAI本文、token、authorization header、full SDK request/responseは記録しない。

- application log group: retention 90日、CloudWatch Logsのservice-managed暗号化、streamはtask ID単位。
- break-glass ECS Exec log group: retention 90日、application logと分離。平常taskからは書き込めない。
- 両log groupへCloudWatch Logs data protection policyを適用し、credential・identifier patternをaudit/maskする。
- UTCで記録し、dashboard表示だけJSTを使用する。
- exceptionは型、code、retryability、stack traceを記録し、secret redaction filterを通す。

## 3. Metrics

namespaceは`ShittimChest/Prod`、dimensionは低cardinalityの`Service`、`Phase`、`Result`に限定する。debate ID、user ID、thread IDをdimensionにしない。

| Metric | Unit | Purpose |
|---|---:|---|
| `BotReadyCount` | Count | READYなBot数、期待値4 |
| `RunningTaskCount` | Count | Container Insightsの稼働task数 |
| `DebateAccepted/Completed/Failed` | Count | 受付と結果 |
| `DebateDuration` | Milliseconds | end-to-end latency |
| `OpenAIRateLimited` | Count | OpenAI 429 |
| `DiscordPublishFailed` | Count | Discord送信失敗 |
| `OutboxPending` / `OldestOutboxAge` | Count/Seconds | 未送信滞留 |
| `LeaseConflict` / `LeaseExpired` | Count | 排他・回復異常 |
| `RecoveryResumed` / `RecoveryFailed` | Count | restart後の再開 |
| `SpotInterruption` | Count | SIGTERM/Spot停止 |
| `DynamoDBWriteFailed` | Count | transaction/conditional write失敗のうち競合以外 |
| `OpenAIRequestCount` | Count | phase/model別API呼出数 |
| `OpenAIInputTokens/OutputTokens` | Count | phase/model別費用分析 |

## 4. Alarm definition

| Alarm | Condition | Period・M-of-N | Missing data | Severity |
|---|---|---|---|---|
| Bot not ready underlying | `BotReadyCount < 4` | 60秒、2/3 | notBreaching | actionなし |
| Task stopped | `RunningTaskCount < 1` | 60秒、2/3 | breaching | warning |
| Running task but Bot unavailable | `RunningTaskCount >= 1 AND Bot not ready underlying` | composite | underlyingに従う | critical |
| Debate failures | `DebateFailed >= 3` | 5分、2/3 | notBreaching | warning |
| Slow debate | `DebateDuration p99 > 300000` | 5分、2/3 | notBreaching | warning |
| OpenAI throttling | `OpenAIRateLimited >= 3` | 60秒、2/3 | notBreaching | warning |
| Discord publishing | `DiscordPublishFailed >= 3` | 60秒、2/3 | notBreaching | critical |
| Outbox stuck | `OldestOutboxAge > 600` | 5分、2/2 | notBreaching | critical |
| Lease contention | `LeaseConflict >= 5` | 5分、1/1 | notBreaching | warning |
| DynamoDB throttle | service metric `> 0` | 60秒、1/3 | ignore | warning |
| DynamoDB write failure | `DynamoDBWriteFailed >= 1` | 60秒、1/3 | notBreaching | critical |

Spot停止中の`Task stopped`は想定事象なのでwarningとし、Bot heartbeat欠損だけでcriticalにしない。taskが稼働しているのに4 BotがREADYでない場合だけcomposite alarmをcriticalとする。Spot停止が24時間以上継続した場合だけ手動escalationする。Alarm actionはdeploy時必須の通知先へ送り、本文・secretを含めない。

## 5. Dashboard

1画面にECS desired/running、Bot READY、debate throughput/result、p50/p95/p99、OpenAI token/429、Discord failure、outbox age、lease、DynamoDB throttle、alarm statusを配置する。deployment annotationにはcommit SHAとimage digestを記録する。

## 6. Health・startup

container health checkはprocessとevent-loop heartbeatだけを検査し、Discord/OpenAI障害でcontainerを再起動ループさせない。新規受付は4 Bot READYかつrepository利用可能の場合だけ有効にする。起動時に設定、Parameter参照、table schema compatibilityを検証し、不足時はfail closedで終了する。recoverable attemptはlease取得後、討論phaseより先にpending outboxをdrainする。未来retry/claim待機中も20秒lease heartbeatを継続し、`discord_outbox_recovered`と`discord_outbox_retry_scheduled`を本文なしで計測する。

## 7. Runbook

### Spot停止・capacity不足

SIGINT/SIGTERMを受けたcallbackは即時に受付と新規interaction dispatchを停止する。readiness monitor、進行中interaction task、startup recovery taskをcancel・awaitし、同一phaseのcheckpoint、controller close、4 Discord client closeをapplication内部90秒以内に行う。ECS側は`stopTimeout=120`とし30秒の終了余裕を残す。次taskは4 READY後に`resume_recoverable`を実行し、lease heartbeat配下でpending outboxをdrainしてからphaseを再開する。shutdown中は新規outbox送信を開始せず、未送信recordを次ownerへ残す。capacity不足中は待機し、on-demandへ切り替えない。

1 Bot切断時は受付を即時停止する。60秒未満の一時切断では進行taskを維持し、60秒連続切断でtaskをcheckpointして待機する。通信断を`FAILED`へ変換せず、4 READY復帰後にfenced leaseを取得して自動resumeする。recovery taskまたはreadiness monitorが例外終了した場合は受付を閉じ、processを異常終了させてECS schedulerの置換対象とする。

checkpoint永続化が失敗した場合はprocess errorとしてalarm対象にするが、Discord clientとsignal handlerのcleanupは継続する。次taskはlease expiryとfencingに従って最後に永続化済みのphaseから再取得し、旧workerのwriteを拒否する。

### Stuck lease

owner、lease expiry、fencing token、最新checkpointを確認する。期限前に手動解除しない。期限後は新workerがconditional updateで取得し、旧tokenのwriteが拒否されることを確認する。

### Outbox不整合

oldest pendingをgeneric Bot slot、nonce、content hash、chunk sequence、Discord message IDで照合する。`next_retry_at`または`claim_expiry`未来値なら期限まで待ち、期限前にclaimを手動解除しない。送信済みなら同一Bot author/nonce/contentのthread履歴を採用してmessage IDを条件付き保存し、未送信なら同じnonceで再送する。同一nonceの内容不一致、locked thread、権限不足は自動retryせずstable error codeとattemptを確認する。DynamoDB recordを直接削除して解消しない。

### Bot token rotation

production taskを停止し、Developer Portalで対象BotのtokenをresetしてSecureStringを更新する。stop-before-start deploy、4 READY、smoke testを確認してから次のBotへ進む。旧新tokenの同時有効を前提にせず、4つを一度にrotateしない。

### OpenAI model利用不能

model availability、project permission、status、SDK error codeを確認する。文書で承認していないmodelへ自動fallbackしない。代替modelは互換性・品質・費用試験後に設定を変更する。

### Discord権限剥奪

対象Guild/channelのView Channel、Send Messages、Create Public Threads、Send Messages in Threadsを確認する。Botを再招待する前にunexpected ownership/config変更がないか監査する。

### DynamoDB PITR

新tableへ35日以内の指定時点で復元し、schema/version/countを検証する。recovery CDK手順でtag、PITR、deletion protection、alarm、IAM resource ARNを再適用し、task roleと参照tableを同一change setで切り替える。production taskを停止して切替え、smoke test後も元tableは原因確定まで保持する。35日より古い状態は復旧対象外とする。

### Failed deploy

CloudFormation eventとECS deploymentを確認し、stateful replacementがあれば停止する。Runtimeは直前の正常digest/task revisionへrollbackし、recovery/outbox smoke testを行う。

### ECS Exec

通常調査で使わず、log/metricで不足する場合だけ承認する。production taskを停止し、write可能root filesystem、Exec agent、shell utility、限定task roleを持つbreak-glass revisionへ切り替える。MFA付き短期credentialで接続し、読取中心、開始・終了・実行理由をincident記録へ残す。調査後は平常のread-only/Exec無効revisionへstop-before-startで戻し、Exec agentがないことを確認する。API callはCloudTrail Event History、session内容は専用90日log groupで監査する。

### GitHub scheduled workflow停止

Public repositoryで60日間activityがない場合、GitHubはscheduled workflowを自動無効化する可能性がある。`dependency-graph.yml`と将来の`drift.yml`を唯一の監視経路とみなさない。月次にActions API/UIでworkflowの`state=active`と直近runを確認し、無効ならmaintainerが再有効化してmanual runを実行する。欠落期間、再有効化時刻、manual run結果を[[20_実装・試験・検証記録]]またはincident記録へ残す。

## 8. Maintenance

- daily: alarm、AWS/OpenAI budget、Cost Anomaly Detection確認。
- weekly: dependency/security PR、ECR finding、CDK drift、failed/recoverable debate確認。
- monthly: cost/SLO、IAM Access Analyzer finding、cost allocation tag、log data protection、runbook連絡先、GitHub scheduled workflowのactive stateと直近runを確認。
- quarterly: token rotation drill、PITR restore drill、Spot interruption drill、least-privilege見直し。
- deploy前: Discord/OpenAI/AWS仕様、model、料金、package version、DNS A/AAAAを再確認する。
