# 実装計画・トレーサビリティ

## 1. Current state

要求仕様、基本設計、監査修正後の詳細設計を確定し、Obsidian正本をGitHub `docs/`へ一方向mirrorする。Public repositoryとmanaged GitHub security/CIは構成済みである。STEP-01〜04BでPython domain/application、GitHub CI、Betterleaks、DynamoDB persistenceを実装・mergeした。STEP-05AはPR `#20`でmerge済みである。STEP-05Bの決定的Question Router、hosted Web search、immutable Evidence、DynamoDB schema v3はlocal実装・試験済みで、PR/CIは未実施である。Responses API Multi-agent betaは使用しない。Terra昇格、Discord adapter、runtime recovery wiring、container、CDK、AWS resource、Discord Applicationは未実装であり、後続sliceごとに別PRとする。

## 3. Implementation sequence

1. Python domain foundation（実装済み）: Python/uv、package build、quality tool、UUIDv7 Debate/Attempt識別子、phase/recovery状態機械、新attempt retry、unit/property test。lock/auditを検証し、結果を[[20_実装・試験・検証記録]]へ残す。
2. GitHub CI（実装済み）: lock、format/lint、strict typing、unit test、audit、package、public-surface、文書/workflowを5つのread-only Public PR checkとして自動化した。PRでCycloneDX source SBOMのstrict schemaと`uv.lock`一致を検証する。Python Dependabot graph jobのmanaged snapshotを使用し、stable mainを確認する週次/manual workflowでGitHub SPDX exportとの一致を確認する。custom Dependency Submissionはmanaged graphに欠落がない限り使用しない。PR `#10`、main CI、managed SBOM照合、Ruleset必須化まで合格した。
3. Secret scanner段階移行（STEP-02B・STEP-02C完了）: BetterleaksをGitleaksと同じ`security` required check内で並行検証し、release provenance、redaction、positive/negative contract、full history、週次latest release監視を確認した。PR `#13`でGitleaksを撤去し、同じcheck名とGitHub managed secret protectionを維持したBetterleaks単独gateへ移行した。main commit `d0f601a`のCI、CodeQL、手動Release Tool Versionsも合格した。
4. Application core（STEP-03完了）: Protocol、accept/run/cancel/retry/resume use case、vote/tie rule、phase/session deadline、TaskGroup cancellation、artifact再利用、unit/property/async test、import-linter gate。PR `#15`をmerge commit `34ccc54`としてsquash merge済みである。
5. Persistence contract（STEP-04A完了）: Guild/channel/operation IDを保持するsnapshot、3-slot `LeaseGrant`、idempotent repository Protocol、DynamoDB native-value縦分割serializer、schema 1→2 up-convert、outbox/panel record、contract test。PR `#16`をmerge commit `54948d7`としてsquash merge済みで、AWS SDKとnetworkを使用しない。
6. Persistence adapter（STEP-04B完了）: boto3 1.43.50 client、accept/retry/terminal transaction、lease acquire/renew/fencing、outbox claim/reschedule/complete、GSI/base pagination、DynamoDB Local 3.3.0とSDK stub test。operation resultは強整合read可能な専用keyを正本とし、10分のSDK tokenやGSIだけへ冪等性を依存しない。PR `#18`をmerge commit `9aafe6e`としてsquash mergeし、PR/main CI、CodeQL、managed Dependency Graphが合格した。
7. OpenAI Structured Outputs adapter（STEP-05A完了）: PR `#20`、merge commit `d6ea561`。stable `responses.parse()`、4 schema、private persona、error変換、usage metric契約。Responses API Multi-agent betaは使用しない。
8. Evidence Router・Web search（STEP-05B local実装済み）: `question-router-v2`、未知表現optional fallback、rules version/reason、none/optional/required、1 Responses request/最大4 tool calls、`web_search_call.action.sources`、immutable Evidence、schema v2→v3、contract test。PR/CIと実API opt-in integrationは未実施。
9. Luna→Terra昇格（STEP-05C候補）: [[02_議論事項・意思決定記録]]でtrigger、phase、budget、deadline、evaluationをoperatorと決定した場合だけ実装する。
10. Discord adapter（STEP-06）: 4 client、command、thread、panel、publisher、contract/integration test。
11. Recovery（STEP-07）: signal、checkpoint、resume、fault injection。
12. Container（STEP-08）: ARM64平常image、break-glass revision、health、SIGTERM test、最終imageのOS/runtime dependencyを含むSPDX SBOM生成。
13. CDK（STEP-09）: TypeScript基盤、Stateful/Runtime/Operations stack、IAM、network、Container Insights、alarm。
14. Release CI/CD（STEP-10）: immutable main/Environment OIDC、単一release workflow、push済みdigestへ別々のbuild provenance/SBOM attestation、manifest、deploy前verify、drift。初回は`bootstrap → Stateful/ECR → image push → Runtime → Operations`の順とする。
15. Operational readiness（STEP-11）: dashboard、runbook、rotation/PITR/Spot drill。
16. Production acceptance（STEP-12）: deploy inputs登録、manual deploy、formal E2E、cost/SLO baseline。

各段階を独立PRにし、先行段階のgateが成功するまで後続のproduction integrationを開始しない。Stateful resourceの作成はapplication/persistence testの合格後とする。

## 4. Deploy-time required inputs

| Input | Rule | Storage |
|---|---|---|
| AWS account ID | operatorがproduction accountを確定 | versioned deploy configまたはrepository variable |
| Guild/channel ID | 1 Guild、1件以上の通常text channel | version付きruntime SecureString |
| Alarm/Budget email | validな通知先、subscription確認必須 | deployment parameter |
| 4 Application ID | Developer Portal作成後にslotへ対応付け | version付きruntime SecureString |
| 4 display name/persona prompt | 3,500 UTF-8 bytes/parameter以下、汎用slotへ対応 | version付きpersona SecureString |
| 4 Bot token | Applicationごとに発行 | SSM SecureString |
| OpenAI API key/project | 専用project、$50 budget | SSM SecureString/config |
| OpenAI model | deploy日に利用可能性・guidance・料金を再確認 | versioned config |

これらは未決設計ではなくdeploy時入力である。token/API keyをObsidianへ記載しない。Guild/channel/Application ID、display name、promptのoperator sourceは公開mirror外だけに置き、runtimeの正はversion付きSSM parameterとする。不足時はdeploy/startupをfail closedにする。

## 5. Definition of done

- Requirement traceabilityの全行に実装PR、test evidence、運用ownerが紐付く。
- CI、integration、formal E2E、fault injection、security、ARM64、CDK試験が合格する。
- Discord Application、AWS/GitHub設定が設計値と一致し、driftがない。
- dashboard、composite alarm通知、PITR、rotation、rollback、break-glass ECS Execのrunbook drillが完了する。
- production cost estimateとBudget notificationを確認する。
- 公式資料をdeploy当日に再確認し、URL、日付、version、差分をPR/ADRへ記録する。
- unresolved critical/high issueがなく、known issueがownerと期限を持つ。
- 各sliceのlocal/CI/deploy試験結果と未実施範囲が[[20_実装・試験・検証記録]]に追記され、Obsidian正本と公開mirrorが一致する。

## 6. Official-source review checklist

詳細設計、実装、review、障害対応の各turnで一次資料を再確認する。AWS Developer Guide/API Reference/Pricing、Discord Developer Docs、OpenAI Developer Docs、Python公式、PyPI、library公式repository、GitHub Docsを使用し、guideとAPI shape/SDK typeを照合する。確認不能なら推測で確定せず作業を保留し、公式挙動との不一致はintegration testとADRへ残す。

## 7. Document responsibility・conflict handling

要求書は要求、意思決定記録は理由と受容risk、各詳細設計はinterfaceとfailure処理、試験設計は合否、本書は対応関係と順序、[[20_実装・試験・検証記録]]は各sliceの実測証跡を担当する。単純な優先順位は設けない。矛盾時は実装を停止し、ADRへ選択肢、影響、決定日を記録して全関係文書を同時更新する。
